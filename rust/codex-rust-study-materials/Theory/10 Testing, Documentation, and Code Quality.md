# 10 Testing, Documentation, and Code Quality

## Почему это часть языка, а не “дополнение”

В Rust инженерная дисциплина встроена в обычный поток разработки:

- тесты запускаются через `cargo test`;
- документация собирается через `cargo doc`;
- форматирование и линтинг - стандартная часть инструментария.

## Unit tests

Их удобно писать рядом с кодом:

```rust
#[cfg(test)]
mod tests {
    use super::*;
}
```

Они хороши для небольших чистых функций и внутренней логики модуля.

## Integration tests

Лежат в каталоге `tests/` и проверяют публичный API как внешний потребитель.

Если unit tests отвечают на вопрос “правильно ли устроен модуль?”, integration
tests отвечают на вопрос “можно ли нормально пользоваться системой снаружи?”.

## Doc tests

Очень сильный инструмент Rust:

```rust
/// ```
/// assert_eq!(2 + 2, 4);
/// ```
```

Документация становится исполняемой и не расходится с кодом так быстро, как
обычный prose.

## Качество кода

Минимальный инженерный цикл:

```bash
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test
```

## Частые ошибки

- Писать тесты только на happy path.
- Вообще не тестировать парсинг ошибок.
- Документировать публичную функцию без примера.
- Считать `clippy` “лишней придиркой”.
