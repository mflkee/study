# 06 Functions, Closures, Traits, Generics, and Lifetimes

## Почему это сильная тема

Именно здесь Rust начинает ощущаться как язык, в котором можно строить
переиспользуемый и расширяемый код, а не только писать маленькие функции.

## Функции и замыкания

Замыкание:

```rust
let curve = |score: u8| (score + 1).min(5);
```

Его удобно передавать в функцию:

```rust
fn adjust_scores<F>(scores: &[u8], f: F) -> Vec<u8>
where
    F: Fn(u8) -> u8,
{
    scores.iter().copied().map(f).collect()
}
```

Важно понимать различия между `Fn`, `FnMut`, `FnOnce`, но на базовом уровне чаще
всего достаточно `Fn`.

## Traits

`trait` задаёт общий интерфейс поведения:

```rust
trait Scorable {
    fn score(&self) -> u32;
}
```

Дальше разные типы могут реализовать этот контракт, и generic-функции смогут
работать с ними одинаково.

## Generics

Обобщение по типу:

```rust
fn best_item<T: Scorable>(items: &[T]) -> Option<&T> {
    // ...
}
```

Это позволяет один раз описать алгоритм и применять его к разным сущностям.

## Lifetimes

Lifetimes не живут отдельно от ownership. Они нужны, когда функция возвращает
ссылку и компилятору надо объяснить, откуда эта ссылка берётся.

Классический пример:

```rust
fn longer<'a>(left: &'a str, right: &'a str) -> &'a str {
    if left.len() >= right.len() { left } else { right }
}
```

Lifetimes чаще всего появляются не потому, что “надо их выучить отдельно”, а
потому что вы начали писать функции, возвращающие ссылки.

## Частые ошибки

- Писать generic там, где он не нужен.
- Бояться trait bounds и поэтому копировать код.
- Пытаться “победить” lifetimes, не понимая, кто владеет данными.
- Считать, что lifetimes всегда надо писать явно. Во многих местах Rust их
  выводит сам.
