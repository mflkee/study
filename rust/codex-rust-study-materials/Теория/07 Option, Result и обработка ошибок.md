# 07 Option, Result и обработка ошибок

## Что нужно перестроить в голове

В Rust ошибка не является чем-то “внезапным и необязательным”. Ошибка - часть
типа и часть контракта функции.

## `Option<T>`

Используйте `Option`, когда отсутствие значения - нормальный сценарий:

```rust
fn next_task() -> Option<String> {
    None
}
```

Типичные методы:

- `map`;
- `and_then`;
- `unwrap_or`;
- `ok_or`.

## `Result<T, E>`

Используйте `Result`, когда операция может завершиться ошибкой:

```rust
fn parse_score(input: &str) -> Result<u8, std::num::ParseIntError> {
    input.parse()
}
```

Оператор `?` позволяет не писать вложенные `match` там, где вы просто хотите
пробросить ошибку выше.

## Собственные ошибки

Для нетривиального кода полезно вводить собственный `enum` ошибок:

```rust
enum ScheduleError {
    MissingParts,
    InvalidTime,
    EmptyTitle,
}
```

Это делает API понятнее и облегчает тестирование.

## Когда `unwrap` допустим

Осторожно и осознанно:

- в одноразовом прототипе;
- в тесте;
- в коде, где вы сами гарантируете корректность входа.

В прикладной логике и библиотечном коде систематический `unwrap` почти всегда
плохой знак.

## Частые ошибки

- Возвращать `Option`, когда нужно различать причины ошибки.
- Возвращать `Result`, когда отсутствие значения - штатный сценарий.
- Скрывать ошибку через `unwrap`, а потом ловить `panic!`.
- Делать слишком общий тип ошибки вроде `String` без причины.
