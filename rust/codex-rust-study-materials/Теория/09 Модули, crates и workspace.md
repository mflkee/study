# 09 Модули, crates и workspace

## Что меняется на этом этапе

До этого момента студент решал отдельные задачи. Теперь важно понять, как строить не просто “код”, а проект.

## Модули

Модуль нужен для разделения ответственности:

```rust
mod domain;
mod storage;
mod report;
```

Хороший модуль отвечает на вопрос “за что этот файл отвечает?”, а не “куда бы ещё деть код”.

## `pub` и границы API

В Rust очень полезно явно думать о публичном API:

- что должно быть доступно снаружи;
- что должно остаться внутренней деталью.

Чем уже и чище API, тем проще тесты и рефакторинг.

## Crates

Иногда выгодно разделить проект на:

- библиотеку (`core`);
- CLI или веб-оболочку (`app`).

Это особенно полезно, когда:

- доменная логика уже заметно выросла;
- её хочется тестировать отдельно;
- интерфейсов запуска может быть несколько.

## Workspace

Workspace собирает несколько crates в один управляемый набор:

```toml
[workspace]
members = ["core", "cli"]
```

Это удобно для курса, если вы хотите:

- держать общий домен отдельно от интерфейса;
- запускать тесты и проверки сразу по нескольким crates;
- показать студенту более взрослую структуру проекта.

## Частые ошибки

- Делить код на модули слишком рано и без смысла.
- Делать один “общий util.rs” на всё подряд.
- Экспортировать слишком много через `pub`.
- Не отделять доменную модель от интерфейсного слоя.
