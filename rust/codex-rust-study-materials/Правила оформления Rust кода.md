# Правила оформления Rust кода

Этот файл не копирует JavaScript-кодстайл из исходного репозитория. Он задаёт практики, которые действительно полезны для учебного Rust-проекта.

## 1. Форматирование

Код должен форматироваться через `cargo fmt`.

```bash
cargo fmt
```

Не спорьте с `rustfmt` на уровне пробелов и переносов строк. В учебном проекте важнее единообразие.

## 2. Линтинг

Перед сдачей задания запускайте:

```bash
cargo clippy --all-targets --all-features -- -D warnings
```

Если `clippy` ругается, сначала попытайтесь понять причину, а не подавить предупреждение атрибутом.

## 3. Именование

- функции и переменные: `snake_case`;
- типы, `struct`, `enum`, `trait`: `CamelCase`;
- константы: `SCREAMING_SNAKE_CASE`.

### Хорошо

```rust
const MAX_RETRIES: usize = 3;

struct StudyTask {
    title: String,
}

fn parse_command(input: &str) -> bool {
    !input.is_empty()
}
```

### Плохо

```rust
const maxRetries: usize = 3;

struct study_task {
    title: String,
}

fn ParseCommand(input: &str) -> bool {
    !input.is_empty()
}
```

## 4. Владение и заимствование

Если функция не обязана владеть данными, не принимайте `String` или `Vec<T>` без причины.

Хорошо

```rust
fn is_valid_title(title: &str) -> bool {
    !title.trim().is_empty()
}
```
Fenced code blocks should be surrounded by blank lines

Плохо

```rust
fn is_valid_title(title: String) -> bool {
    !title.trim().is_empty()
}
```

## 5. Обработка ошибок

В библиотечном коде избегайте `unwrap()` и `expect()`, если ошибка может возникнуть на реальных входных данных.

Хорошо

```rust
fn parse_score(input: &str) -> Result<u8, std::num::ParseIntError> {
    input.parse()
}
```

Плохо

```rust
fn parse_score(input: &str) -> u8 {
    input.parse().unwrap()
}
```

Допустимое исключение: учебные тесты, одноразовые прототипы или код `main`, где вы сознательно завершаете программу с сообщением.

## 6. `enum` лучше набора магических строк

Если сущность имеет конечный набор состояний, используйте `enum`.

Хорошо

```rust
enum TaskStatus {
    Todo,
    InProgress,
    Done,
}
```

Плохо

```rust
struct Task {
    status: String,
}
```

## 7. `match` должен быть осмысленным

Если вы используете `match`, обрабатывайте варианты явно. Не прячьте логику в цепочку неясных `if`.

## 8. Небольшие функции

Функция должна решать одну задачу. Если в ней одновременно:

- парсинг входа;
- изменение состояния;
- форматирование ответа;
- работа с файлами;

то её почти наверняка надо разделить.

## 9. Тесты обязательны для ключевой логики

Минимум:

- happy path;
- один-два краевых случая;
- один негативный сценарий с ошибкой.

Если функция чистая и короткая, почти всегда проще написать тест, чем объяснять потом, почему код “должен работать”.

## 10. Комментарии нужны редко

Не пишите комментарии, которые дублируют код.

Плохо

```rust
// Увеличиваем счётчик на 1
count += 1;
```

Пишите комментарий только там, где без него трудно понять инженерное решение.

## 11. Предпочитайте читаемость “умности”

Для учебного кода лучше простой `for` и понятная структура данных, чем запутанная цепочка адаптеров итератора ради краткости.

## 12. Финальная проверка перед сдачей

```bash
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test
```
